# Ralph Dev（完整开发闭环）

本技能定义**从分支与需求到 CI 全绿**的完整流程：先保证不在默认分支上直接改代码；再**完整执行 feature-dev 技能**（见第 1 节，不得以本节摘要代替该技能正文）；用测试与（如有 UI）浏览器验收把关「开发完成」；最后提交、PR 或 push，并**监控 CI 直至通过**。

## 原则

- **未完成 CI 闭环不得宣告任务结束**：在 PR 的检查未达「可合并/全绿」前不得宣称任务完成。
- **先澄清再大改**：架构设计与实现需用户确认后再动手；细则以 **feature-dev** 技能为准。
- **证据优先**：本地测试、浏览器结果、`gh pr checks` 输出作为结论依据。
- **硬门禁不可省略**：`§1 feature-dev` 与 `§3 提交/PR/CI` 均为强制流程；除非用户明确书面豁免，否则不得跳过、压缩成口头说明、或以“时间原因”省略。
- **本技能是外层编排**：**feature-dev** 是其中嵌套的一段流程。在 feature-dev 里向用户提问、等待答复、等待架构拍板，**只是暂停点，不是任务终点**；除非用户明确改口取消整次交付，否则**必须**在用户回复后继续执行本技能 **§2 → §3 → §4**，直至满足第 4 节的完成定义。

## 编排与多轮对话（避免停在 feature-dev）

多轮对话里模型容易在「答完用户、做完 feature-dev 某一阶段」后提前收尾。**按下面做可避免烂尾：**

1. **启用本技能时立刻建一条贯穿全线的 Todo**（或用下方「清单」逐项对应）：§0 分支 → §1 feature-dev（可细分子阶段）→ §2 质量门禁 → §3 提交/PR/CI → §4 汇报。在 **§4 完成前**不得把所有 Todo 标成完成，也不得用「本次对话先到这里」结束。
2. **因 feature-dev 必须等待用户时**：回复末尾**简短声明**仍在执行 **ralph-dev**，并列出**未完成的章节**（例如：「待你确认架构后，我会继续实现 → §2 测试与浏览器验收 → §3 提交与盯 CI」）。不要让人误以为 feature-dev 结束即整单结束。
3. **用户再次发消息后**：先根据上条承诺对齐进度，从**第一个未完成的 § 或小节**继续，**不要**重新从 §0 开始，除非上下文丢光或用户要求重来。
4. **feature-dev 的 Phase 7 Summary** 只总结「需求到实现/评审」这一段；**不能**替代本技能的收尾——Phase 7 之后仍须执行 **§2**（若实现阶段尚未跑过门禁）、**§3、§4**。
5. **唯一可以提前结束的情况**：用户明确说只要设计/只要到 PR 前、或取消任务；否则默认要走到 **CI 全绿 + 第 4 节汇报**。

## 0. 分支门禁

1. 解析当前分支：`git branch --show-current`（或 `git rev-parse --abbrev-ref HEAD`）。
2. 解析仓库默认分支：例如 `git symbolic-ref refs/remotes/origin/HEAD | sed 's@^refs/remotes/origin/@@'`，或 `git remote show origin | sed -n '/HEAD branch/s/.*: //p'`。
3. 若当前分支是默认分支（常见为 `main` / `master`）且用户要做功能开发或修复：
   - **不要**在默认分支上直接开发。
   - 创建并切换到功能分支：`git checkout -b <type>/<short-description>`（如 `feat/add-login`、`fix/ci-flake`）。分支名结合任务含义；不确定时简短询问用户。
4. 若已在非默认分支，继续下一步。

紧急热修若用户明确要求在默认分支操作，则遵从用户指令并注明风险。

## 1. 按 **feature-dev** 技能执行（需求到设计）

**必须**在动手实现前加载并遵循 **feature-dev** 技能的完整说明（例如本仓库或其它技能目录下的 `feature-dev/SKILL.md`，以及你在 Cursor 中配置的同名技能；若用户通过 **`/feature-dev`** 挂载了等价指令，则以该指令对应的**完整**内容为权威）。执行时按该技能所写的阶段、子任务、Agent/工具用法与门禁逐项落实。

### 1.1 严格执行约束（防止“没走完整 feature-dev”）

- **禁止仅引用摘要**：不得只复述 feature-dev 的阶段名后直接开工；必须按 feature-dev 原文要求逐阶段推进。
- **禁止越级进入实现**：若仍有未澄清问题（Phase 3）或架构未获用户明确确认（Phase 4），不得进入 Phase 5 Implementation。
- **禁止“默认同意”**：用户未明确拍板架构前，不得把“无回复/看起来可行”当作同意。
- **每次暂停要报待办**：因等待用户而暂停时，明确列出“当前停在 feature-dev 的哪一阶段、下一步恢复点、恢复后要继续到 §2/§3/§4”。
- **恢复后续跑**：用户一旦回复，必须从上次暂停点继续，不得重置流程，也不得把 feature-dev 视作“已自然结束”。

下表**仅作阶段索引**，便于扫一眼进度；**不得**用表格替代 feature-dev 正文里的详细要求（例如并行 code-explorer、澄清问题列表、架构多方案与用户拍板、实现前等待批准、多视角 code review 等）。

| 阶段 | 目标（索引，细节以 feature-dev 为准） |
|------|------|
| Phase 1 Discovery | 理解问题、范围、成功标准；必要时向用户提问。 |
| Phase 2 Exploration | 并行阅读/探索相关代码，列出关键文件与模式。 |
| Phase 3 Clarifying Questions | **列清所有待确认点，等待用户答复后再设计。** |
| Phase 4 Architecture | 可给出多方案与推荐；**待用户选定或明确同意后再实现。** |
| Phase 5 Implementation | 按选定方案实现；用 Todo 跟踪进度。 |
| Phase 6 Quality Review | 可做代码审查与问题汇总，按用户意见处理。 |
| Phase 7 Summary | 说明改动、决策与后续建议。 |

何时可跳过或压缩步骤：**仅当 feature-dev 技能自身允许，或**前序对话已完整走完等价流程且用户确认无需重复——即便如此也应简短复述共识点，再进入实现。

## 2. 开发完成的质量门禁（代码可交付）

在进入「提交 / PR」之前须满足：

1. **单测**：为新逻辑或修复补充/更新单元测试；全部通过。
2. **集成测试**：按项目惯例运行集成测试；全部通过。若项目无集成测试约定，则运行最接近的端到端/契约测试或文档中规定的检查。
3. **有页面的功能**：在本地（或指定环境）**实际启动应用**。验收**必须优先**使用 **agent-browser**：先加载并遵循 **agent-browser** 技能（用户侧常通过 **`/agent-browser`** 引用），再用 CLI `agent-browser` 按「`open` URL → `snapshot -i` 拿 refs → `click` / `fill` 等交互 → 导航或 DOM 变化后**务必重新 snapshot**」对照预期走关键路径、交互、可见文案/布局；布局或像素级确认可用 `screenshot` / `--annotate`。**不要**默认改用其它浏览器工具；仅当 agent-browser 不可用时，再退而求其次使用 Cursor 浏览器 MCP 等等价能力，并在汇报里写明原因。不一致则修到一致再进入下一步。
4. 运行项目 `CLAUDE.md` / `AGENTS.md` 中要求的其它检查（lint、类型检查等）。

**以上任一项未满足，不视为开发完成**，不得进入第 3 节。

## 3. 提交、PR / Push 与 CI 闭环

开发完成且质量门禁通过后，按顺序执行（非 GitHub 时用平台等价 CLI/Web）：

### 3.0 不可中断规则（防止“提交后流程没走完”）

- `git commit` 成功**不代表结束**；提交后仍必须继续执行 **push → PR（创建或复用）→ CI 监控 → 失败修复循环**。
- 若用户目标是“完整交付/合并就绪”，则在 CI 达到可合并状态前不得停止在“我已提交代码”。
- 只有两种合法停止条件：  
  1) CI 全绿（或团队认可的等价可合并状态）；  
  2) 用户明确要求到某一步即止（例如“先只提交不提 PR”）。

### 3.1 前置

- 已安装并登录 GitHub CLI：`gh auth status` 成功；`origin` 指向 GitHub。

### 3.2 提交与推送

1. `git status`；按需 `git commit`（清晰提交信息，遵循项目约定）。
2. `git push -u origin HEAD`（已设 upstream 则 `git push`）。

### 3.3 是否存在 PR

当前分支下：

```bash
gh pr view --json number,url,title,state 2>/dev/null
```

- 有 JSON 输出：已有 PR，记下 `url`，进入 3.4。
- 无 PR：执行 `gh pr create --fill`（或按项目要求补 title/body/base），再进入 3.4。

### 3.4 监控 CI

```bash
gh pr checks --watch --interval 10
```

说明：`gh pr checks` 默认对**当前分支关联的 PR** 生效；`--watch` 会阻塞至检查结束；与分支保护一致时可加 `--required`。若 watch 不可用，可用 `gh pr checks --json name,state,bucket,link` 轮询，直到无 `pending` 且失败为空。

**禁止**：未看到绿色检查（或用户明确接受的例外）前写「已完成」「CI 应该过了」等结论。

若有检查失败：进入 3.5，修好后重复 3.2→3.4，直至通过。

### 3.5 CI 失败时的修复循环

1. 从 `gh pr checks`、失败 job 日志（`gh run view` / Actions 网页）取证，避免臆测。
2. 修复后本地能复现的检查先跑过；再 `git commit` → `git push`。
3. 回到 3.4，直至通过。

## 4. 任务完成定义（Definition of Done）

同时满足：

- 第 2 节质量门禁全部满足。
- 第 3 节已执行，且 PR 上**所需检查均已通过**（或与团队约定一致的可合并状态）。
- 向用户汇报：分支名、PR 链接、CI 结论、关键测试与（若适用）浏览器验收摘要。

**反例（均不算完成）**：

- 只完成了 feature-dev 讨论或实现，但未跑 §2 质量门禁。
- 只做到 `git commit` / `git push`，未创建/检查 PR、未监控 CI 到结束。
- CI 仍在 pending / failing，却给出“已完成”结论。

## 与其它技能的配合

- **agent-browser**：有 UI 时的**首选**验收工具（见 §2 第 3 条）；与 agent-browser 技能中的快照/交互/等待/截图规范一致。
- **systematic-debugging**：CI 或本地失败时先取证再改。
- **verification-before-completion**：宣称完成前用命令输出证明测试与 CI 状态。
